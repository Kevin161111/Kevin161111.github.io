程序员的思维模式

发表于 2017-03-04 | 分类于 其他 |

程序员的思维框架-DRY(Don’t Repeat Yourself)

今天和小伙伴Meetup，曹望给我展示了程序员是怎么想问题的。

在网站的商品展示页面显示其他网站的图片，这个图片是放在其他网站的服务器上面的。在他的网站上能够实现。而且不用自己上传，只需要后台自己调用就好了。

  1. 用python scrapy框架爬虫你要找的网站商品。scrapy框架
  2. 分析目标网站的分布及内容，加入到scrpy框架中
  3. 爬出的内容按照你想要的内容转为.json数据库
  4. 通过seed文档提取.json数据库的内容
  5. 新建产品图片及说明的表单，栏位设置要符合想要的内容
  6. 在controller 中调出来
  7. 在view实现

这个你就不用一个个上传产品了，也不用在图片放到aws的服务器了，如此这般快速，就是把重复的事情用代码来实现。别的就不感慨了，太神奇！

工具：navicat premium，数据库可视化管理软件

远程协作的框架

发表于 2017-03-04 | 分类于 编程 |

设定远程公共分支dev

两人约定共同的分支,名字为dev，将其他分支和功能都合并到这个分支里面，保持一个完整的功能

远程分支拉到本地

push pull origin dev,这样你在本地就可以实现所有的功能。

创建本地功能分支b1（采用个人名+实现功能的命名方式）

github是鼓励一个完整功能创建一个分支的，之后执行git mr dev，这样本地b1就有了所有的功能，在此基础上进行代码添加

将本地b1推到github

实现远程repo中有(个人名+实现功能的命名)这个分支，方便检查。

合并功能

  1. git checkout dev
  2. git mr b1
  3. git push origin dev

远程合并

这样就把本地的b1上传到dev了，你的同伴重复这样的操作就把他的功能添加这个专案了，你再次做的时候就重新建立一个分支，把dev 拉下来重复上面的步骤就好了。

什么是userstory？

发表于 2017-02-20 | 分类于 编程 |

userstory是工程师一个很重要的能力，站在用户的角度，用简洁的语言描述一个功能。
为什么这个能力很重要？从实际的需求转换成实际的功能是通过它实现的，功能到实际的代码也是用它来做出来的
这是工程师思维中很重要的一个存在。也是后续用它来进行项目管理的一个目标。

严谨、漂亮、详细都不是他的要求，严谨的文件本身是一个目标而不是开放的功能，语言本身就是模糊的，用详细的文件去描绘会造成“确定性的幻想”以为规划好了就无需沟通了，导致最终做出不符合预期的功能。

实例：

  1. 身为消费者，我要很方便的在这个网络商店买到东西
  2. 在前台买到东西并结账
  3. 加入购物车，生产订单
  4. 生成订单后，填写地址并付款
  5. 结账后，生成订单确认信件
  6. 店家收到已经付款的订单通知
  7. 货架上的库存必须随之减少
